How Canonical Truth Flows

Acepe's canonical architecture is organised around one rule: product truth is Rust-owned. Provider output is input. TypeScript and @acepe/ui are read-only consumers. When something looks wrong in the agent panel, the fix belongs upstream — never in a component.

The core pipeline

Provider (Claude, Cursor, Codex, …) │ ▼ Rust adapter / history parser ← provider quirks live HERE only │ ▼ Canonical model (Rust-owned truth) • SessionStateGraph • CanonicalTranscriptEvent • OperationSnapshot • InteractionSnapshot │ ▼ Envelopes → TypeScript projection ← read only, no fallbacks │ ▼ Scene model (pure mapper) ← agent-panel-graph-materializer │ ▼ View (@acepe/ui) ← presentational only

This is the "GOD" (Green) architecture gate. The glossary lives in CONTEXT.md; the enforcement checklist is in the GOD Architecture Check skill.

Six authority surfaces

Think of the canonical model as six surfaces, each with a single owner. Consumers may project and render — never repair.

SurfaceWhat it owns
SessionStateGraph Lifecycle, activity, turn state, capabilities, revision. Composed in session_state_engine/graph.rs; runtime registry in runtime_registry.rs.
Canonical transcript Strict event order, display ids, user/assistant/tool rows. Built from CanonicalTranscriptEvent in transcript_projection/.
Operation graph Tool call lifecycle — args, results, parent/child. Linked to transcript via OperationSourceLink::TranscriptLinked.
Interaction graph Permissions, questions, blocking user prompts awaiting reply.
Tool identity authority Single funnel for tool kind and name: acp/reconciler/. No parallel classifiers in parsers or UI.
Transient UI state Animation, click guards, pending send intent — TypeScript only, and only if it is not product truth.

Where the Rust modules live

Under packages/desktop/src-tauri/src/acp/, the canonical spine clusters like this:

acp/
├── providers/<name>/     # one home per provider (live + history ingress)
├── parsers/              # provider row → semantic blocks
├── reconciler/           # tool identity authority
├── transcript_projection/# CanonicalTranscriptEvent, display_id, snapshots
├── session_state_engine/ # SessionStateGraph, envelopes, ledgers
├── session_materialization/ # transcript + operations assembly
├── session_open_snapshot/   # SessionOpenResult at open time
└── transcript_viewport/  # scroll/viewport projection (DOM-adjacent)

Provider data enters at the edges (providers/, parsers/, history parsers in cursor_history/, session_jsonl/). It must be normalized before it crosses into transcript_projection or session_state_engine.

What crosses the Rust → TypeScript boundary

Rust emits session state envelopes (SessionStateEnvelope). TypeScript applies them through the envelope router (session-state-command-router.ts) into CanonicalSessionProjection — a read-only mirror of SessionStateGraph.

The agent panel then uses the MVC split defined in CONTEXT.md:

  • Controller — agent-panel.svelte: reads stores, routes actions
  • Model — agent-panel-graph-materializer.ts: maps canonical facts → scene entries
  • View — @acepe/ui agent-panel components: props in, render out
The Green Rule: If you write canonical != null ? canonical.X : hotState.X, stop. Fix the canonical producer instead.

Identity: order vs metadata

A common canonical bug is treating Claude's message.id as a display id. Multiple JSONL rows can share the same message.id (text row + tool rows). Canonical order comes from transcript_seq; UI identity from Acepe-owned display_id; tool identity from tool_call_id.

// CanonicalTranscriptEvent (simplified)
transcript_seq    // order authority
display_id        // Acepe-owned UI identity
provider_msg_id   // metadata (e.g. msg_abc)
tool_call_id      // tool identity (toolu_*)
kind              // UserText | AssistantText | ToolUse | …
