Rust Adapter / History Parser

In Lesson 1 you learned that provider output is input, not product truth. The Rust adapter / history parser is the membrane where that input becomes canonical facts. Every provider quirk Acepe tolerates must live here — and nowhere else.

Adapter vs parser — two names, one edge

CONTEXT.md uses both terms for the same boundary:

  • Adapter — the provider module home under acp/providers/<name>/ (transport, settings, enrichment, edge map).
  • Parser — the code that turns raw bytes into semantic structures (acp/parsers/ for live; cursor_history/, session_jsonl/ for disk).

Think of the adapter as the address book (where each ingress edge lives) and the parser as the translator (what each edge does).

Two ingress modes

Every provider hits Acepe through one of two doors:

┌─────────────────────────────────────────────────────────┐ │ LIVE PARSE (session connected) │ │ ACP JSON updates → AgentParser → SessionUpdate │ │ → session_state_engine / streaming accumulator │ └─────────────────────────────────────────────────────────┘ ┌─────────────────────────────────────────────────────────┐ │ HISTORY RESTORE (session open / import) │ │ Disk files → history parser → FullSession │ │ → session_converter → CanonicalTranscriptEvent │ │ → session_materialization → TranscriptSnapshot │ └─────────────────────────────────────────────────────────┘

Both paths must produce compatible canonical facts — same tool call ids, same order semantics, same display id inputs. Cursor documents this explicitly in its ingress edge map (acp/providers/cursor/mod.rs) with tests in parsers/tests/cursor_ingress_edges.rs.

Live path: AgentParser

When a session is connected, ACP delivers streaming JSON updates. Each provider implements the AgentParser trait (acp/parsers/types.rs):

pub trait AgentParser: Send + Sync {
    fn agent_type(&self) -> AgentType;
    fn detect_update_type(&self, data: &Value) -> Result<UpdateType, ParseError>;
    fn parse_tool_call(&self, data: &Value) -> Result<ToolCallData, ParseError>;
    fn parse_tool_call_update(&self, ...) -> Result<ToolCallUpdateData, ParseError>;
    fn detect_tool_kind(&self, name: &str) -> ToolKind;  // hint, not authority
    // …questions, todos, usage telemetry, plans
}

Dispatch is via get_parser(AgentType::Cursor) (etc.). Parsers normalize update shapes — e.g. toolCall vs tool_use — into shared SessionUpdate variants the session state engine already understands.

Important: detect_tool_kind on the parser is a parse-time hint. The single authority for tool identity is acp/reconciler/ (classify_raw_tool_call, etc.). Parsers must not become parallel classifiers.

History path: disk → FullSession → canonical events

When you open a past session, Acepe reads provider files from disk. Each provider has its own on-disk format:

ProviderDisk location (typical)Parser module
Claude Code ~/.claude/projects/…/sessions/*.jsonl session_jsonl/parser/full_session.rs
Cursor ~/.cursor/projects/…/agent-transcripts/, store.db cursor_history/parser.rs, history/cursor_sqlite_parser.rs
Copilot Copilot session state files copilot_history/parser.rs
OpenCode Server history API opencode_history/parser.rs

History parsers deserialize provider rows into a shared intermediate: FullSession (session_jsonl/types.rs) — an ordered list of OrderedMessage with ContentBlock variants (text, tool use, tool result, etc.).

session_converter: the shared funnel

Once you have a FullSession, provider-specific conversion diverges then reunites:

session_converter/
├── claude.rs      # Claude-specific FullSession → thread snapshot
├── cursor.rs      # Cursor-specific paths + tool id normalization
├── fullsession.rs # Shared: StoredEntry → CanonicalTranscriptEvent
└── transcript_events.rs  # Order authority lives here

materialize_canonical_transcript_events in transcript_events.rs is where history transcript order is decided. It walks messages, groups compatible assistant fragments, assigns transcript_seq, and dedupes tool rows. This is the fix point for bugs like "tool calls appear after assistant text when they shouldn't."

session_materialization: join transcript + operations

After conversion, session_materialization/ assembles the full canonical picture:

  1. Build TranscriptSnapshot from canonical events (or stored entries as fallback).
  2. Project operations and interactions via ProjectionRegistry.
  3. relink_operations_to_transcript — link each operation to its transcript entry by tool_call_id.
  4. ensure_transcript_tool_operations — every tool row in the transcript has a matching operation.

The transcript is the authority for entry identity. Operations link to it — never the reverse.

Provider module homes (edge maps)

Each provider's mod.rs documents its ingress edges. Example from Claude Code:

// acp/providers/claude_code/mod.rs — ingress edge map
// Live parse    → acp/parsers/claude_code_parser.rs
// History restore → session_converter/ Claude paths
// Classification  → acp/reconciler/providers/claude_code.rs

Before editing provider behavior, read the edge map. If your change touches tool ids, check that live parse and history restore still agree (Cursor's cursor_ingress_edges.rs tests are the pattern).
